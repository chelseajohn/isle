1 \page filelayout File Layout
2 
3 Almost all components of Isle use HDF5 files with a common layout to store data.
4 Many functions are flexible though and can read from files that have slightly different layouts.
5 In the interest of compatibility however, the default layout should be used whenever possible.
6

8 ## HDF5 Files
9 
10 With the exception of lattices (see below), all data is stored in HDF5 files.
11 Isle uses two kinds of such files, *configuration* and *measurement* files.
12 Those kinds are compatible and it is possible to store all data in a single file.
13 
14 ### Configuration Files
15 
16 Configuration files always contain three groups at the root level:
17 - <B>`configuration`</B>: Stores field configurations obtained from HMC.
18  The group contains additional groups labeled by the trajectory index.
19  Each of those groups contains three fields
20  * `phi`: The field configuration. (Complex lattice vector)
21  * `action`: Value of the action for that configuration. (complex number)
22  * `trajPoint`: Point along the trajectory that was accepted. (integer)
23 - <B>`checkpoint`</B>: Stores checkpoints along the Markov chain to resume HMC runs.
24  The group contains additional groups labeled by the trajectory index.
25  Note that there can be fewer checkpoints than configurations.
26  Each of those groups contains three fields
27  * `cfg`: The field configuration. (Link to sub group `/configuration`)
28  * `rng_state`: State of the RNG at the end of the trajectory.
29  The content depends on the specific RNG but there is
30  always a dataset called `name` which identifies the RNG.
31  (HDF5 group)
32  * `evolver`: A group containing the state of the evolver at the end of
33  the trajectory. See below for more information. (HDF5 group)
34 - <B>`meta`</B>: Stores information on the physical system and algorithmic parameters.
35  Contains the following objects:
36  * `action`: Source code of a function that instantiates the action which was
37  used to generate the configurations in the file. (String)
38  * `lattice`: YAML representation of the lattice, see below. (String)
39  * `params`: YAML representation of the parameters dataclass object.
40  Stores user defined parameters, physical and otherwise. (String)
41  * `evolvers`: General information on evolvers, see below. (Group)
42  * `version`: Group containing various versions of the software that was used
43  to make the file, most notably the version of Isle.
44  All versions are stored as strings.
45 
46 #### Evolvers
47 
48 Evolvers are stored in two different places in HDF5 files.
49 The `/meta/evolvers` group contains an enumerated list of the types of evolvers
50 and `/checkpoint/<itr>/evolver` contains the state of the evolver after the
51 corresponding HMC trajectory.
52 
53 The types of evolvers are stored so they can be reconstructed at a later point in time
54 to, for instance, continue an HMC run.
55 Types are saved on demand only; as long as no checkpoints are written, no types are saved.<br>
56 There are two ways, the types can be stored:
57 - If the evolver is built into Isle or passed to the EvolverManager instance via the
58  `definitions` parameter (e.g. through the parameter `definitions` of `isle.drivers.meas.newRun`),
59  only its name is stored in a dataset called `__name__`.
60 - If the evolver is custom defined and not included in `definitions`, the full source code of the
61  evolver's definition is stored and `__name__` is set to `"__as_source__"`.
62  The dataset `__source__` contains the full source code of the evolver.
63 
64 The group of the evolver state contains an attribute `__index__` which holds an integer
65 indicating which entry in the types group contains the evolver's type.
66 Any other content is determined by the evolver itself.
67 Evolvers are allowed to refer to the type definitions.
68 
69 \see Additional information on %evolvers, in particular handling of types,
70 can be found in \ref evolversdoc.
71 
72 ### Measurement Files
73 
74 Measurement files contain the same metadata at the root level as configuration files.
75 In addition, the results of measurements are written to the root level but each measurement
76 writes into its own separate group.
77 All group names depend on the user and can be chosen when calling the measurement driver.
78 The group contents are determined by the measurements.
79 
80 The only common element is an attribute to the group called `configurations`.
81 This attribute contains a string in slice notation (`start:stop:step`) indicating which
82 configurations the measurement was taken on.
83 The indices in those slices are interpreted relative to the trajectory indices (group names)
84 of configurations in the `/configuration` group.
85

87

89 ```

91 ├─ configuration
92 │ ├─ 2
93 │ │ ├─ phi (complex lattice vector)
94 │ │ ├─ action (complex number)
95 │ │ └─ trajPoint (integer)
96 │ │
97 │ ├─ 4
98 │ │ ├─ phi (complex lattice vector)
99 │ │ ├─ action (complex number)
100 │ │ └─ trajPoint (integer)
101 │ ┆
102 │
103 ├─ checkpoint
104 │ ├─ 2
105 │ │ ├─ cfg (link to /configuration/0)
106 │ │ ├─ rng_state (group, RNG dependent)
107 │ │ └─ evolver (group, evolver dependent)
108 │ ┆
109 │
110 ├─ meta
111 │ ├─ action (source of function)
112 │ ├─ lattice (YAML)
113 │ ├─ params (YAML)
114 │ ├─ evolvers (group)
115 │ └─ versions (group)
116 │
117 │ # measurements:
118 │
119 ├─ action
120 │ ├─ [configurations] (string attribute)

122 │
123 ├─ correlation_functions
124 │ ├─ single_particle

136 ## Lattice Files
137 
138 Lattices are stored as [YAML](https://yaml.org/).
139 Those YAML strings are by default stored in stand alone files (/resources/lattices) or as part
140 of the metadata in HDF5 files (see above).
141 
142 Lattices are encoded via custom YAML nodes denoted by `!lattice`.
143 These nodes contain a mapping with the following keys:
144 - <B>`name`</B>: A string identifying the lattice. Should be unique to avoid conflicts.
145 - <B>`comment`</B>: An arbitrary string to give detailed information on the lattice.
146 - <B>`adjacency`</B>: A list of lists describing the adjacency matrix.
147  Each element of the outer list is a list of two integers indicating that
148  those two sites are connected.
149 - <B>`hopping`</B>: Can be either
150  * A float: The hopping strength for all connections in the lattice.
151  * A list of floats: Hopping strength for each connection. Must be the same length
152  as `adjacency` and is specified in the same order.
153 - <B>`positions`</B>: List of lists containing the 3D positions of all lattice sites.
154  Each element of the outer list is a list of three floats, the 3D
155  coordinates of that site.<br>
156  This field determines how many lattice sites there are and which order
157  they are indexed in.
158  Both `adjacency` and `hopping` have to be consistent with that.
159 - <B>`nt`</B>: The number of time slices.
160  This field is *optional* and defaults to 0.
161  Stand alone files should not set `nt` but only describe the
162  spatial properties of the lattice.
163  This allows those files to be reused easily but users have to make sure
164  to set `nt` manually after reading a lattice from YAML.
165 
166 Here is an example of a lattice with four sites in a hexagonal alignment.
167 This is taken from /resources/lattices/four_sites.yml.<br>
168 In this lattice, site 0 is connected to site 1 with hopping strength 1 and to
169 site 3 with strength 2, etc.
170 The position of site 0 is `(0, 0, 0)`, of site 1 is `(1, 0, 0)`, etc.
171 
172 ```{.yml}
173 !lattice
174 name: 'four sites'
175 comment: 'Four sites (hexagonal)'
176 adjacency:
177  - [0, 1]
178  - [0, 3]
179  - [1, 2]
180  - [2, 3]
181 hopping: [1, 2, 2, 1]
182 positions:
183  - [0, 0, 0]
184  - [1, 0, 0]
185  - [1.5, 0.866025403784439, 0]
186  - [2.5, 0.866025403784439, 0]
187 nt: 32
188 ```
189 
190 \see isle.yamlio.loadLattice for a convenient way to read lattices from YAML files.<br>
191  Thanks to the custom node, lattices can be loaded via `yaml.safe_load(lattice_string)`
192  and saved via `yaml.dump(lattice)`.
